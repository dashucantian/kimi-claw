# 卡 搭-7：用AI助手开发「止止」小程序——数字时代的心智锚点

## 场景故事

周五下午，宗国盯着电脑屏幕上的微信开发者工具。三天前，他还不会写一行小程序代码。

藏地八年的修行，三年岩洞的闭关，让他在重庆武陵禅寺推动"禅宗修道班"时看清了一个现实：现代人太需要止观训练，却挤不出完整的时间进山。等电梯的三十秒、会议间隙的五分钟、睡前的十分钟——这些碎片时间如果能被利用起来，或许比周末去一趟寺庙更有用。

他想做一个工具。不是让人逃离数字生活，而是教人在碎片中练习收放。

但1998年的科技情结早已生疏。如果走传统开发路线，找外包、写需求、反复沟通，这个念头可能永远停留在"想法"阶段。

于是他换了一条路：把AI当作结对编程的伙伴。

第一天，他花了一整天和AI对话，把"四阶段禅修流程"翻译成代码结构。第二天，音频播放的bug修到深夜。第三天，当Edge TTS生成的引导语音第一次在手机上流畅播放时，他知道这条路走通了。

一个月后，「止止」小程序完整上线。调身调息→止息→观心→收功，四个阶段环环相扣。用户开始留下反馈："等电梯的时候用了，真的有用。"

这不是"不会代码的人用AI生成代码"的励志故事。这是一个有清晰愿景的人，用AI把愿景翻译成可执行方案的过程。AI写的每一行代码，都经过他的验收和调整。他负责"要做什么"和"对不对"，AI负责"怎么做"和"怎么写"。

## 八格表单

### 1. 什么时候用？（触发时刻）

- 有产品/工具想法，但不会写代码或不想从零开始写
- 想快速验证一个MVP，而不是先做三个月需求文档
- 已有明确场景和用户，缺的是技术实现路径
- 【待补充：其他触发场景】

### 2. 做出来是什么？给谁？（目标产出）

| 项目 | 说明 |
|------|------|
| **格式** | 可运行的微信小程序（或其他平台应用） |
| **内容** | 核心功能完整、UI可用、能上线测试 |
| **给谁** | 目标用户测试、投资人演示、自我验证 |

### 3. 需要准备什么？（输入清单）

| 类型 | 内容 | 说明 |
|------|------|------|
| **必须** | 产品愿景描述 | 解决什么问题、给谁用、核心场景 |
| **必须** | 功能清单（MVP版） | 砍掉一切非核心，只留"没有就不成立"的功能 |
| **建议** | 参考案例 | 类似产品截图或交互描述 |
| **建议** | 技术偏好 | 微信小程序/网页/APP？有倾向就明说 |
| **可选** | 设计稿/草图 | 手绘线框也行，让AI理解你的审美方向 |

**关键点**：不需要准备好技术方案，描述清楚"要什么"就行。实现路径是AI的活。

### 4. 怎么跟AI说？（提示词模板）

```markdown
## 背景
我是【宗国】，正在开发一个【微信小程序】。
产品定位：【融合正念冥想与AI技术的数字修行工具】
目标用户：【想在碎片时间练习冥想的都市人群】

## 核心功能（MVP）
1. 【四阶段禅修流程】：调身调息 → 止息 → 观心 → 收功
2. 【AI语音引导】：基于Edge TTS的动态音频生成
3. 【练习记录】：数据追踪与进度可视化

## 技术约束
- 平台：微信小程序
- 音频：使用Edge TTS生成引导语音
- 数据：本地存储优先，必要时云同步

## 任务
请帮我实现【具体功能模块】，输出完整可运行的代码。

## 输出要求
1. 完整代码文件（wxml/wxss/js/json）
2. 关键逻辑的注释说明
3. 测试步骤和验收标准
4. 已知限制和待优化项

## 约束条件
- 代码需符合微信小程序规范
- 音频播放需处理微信小程序的自动播放限制
- 阶段性流程需有清晰的开始/结束/跳转逻辑
```

### 5. 具体怎么操作？（步骤拆解）

| 步骤 | 动作 | 耗时 |
|------|------|------|
| 1 | 写清楚产品愿景和功能清单 | 15分钟 |
| 2 | 选择第一个要实现的模块 | 5分钟 |
| 3 | 填入模板，发送给AI | 2分钟 |
| 4 | AI生成代码，你本地运行测试 | 10分钟 |
| 5 | 记录问题，反馈给AI迭代 | 5分钟 |
| 6 | 验收通过，进入下一个模块 | 5分钟 |
| **合计** | | **约30-40分钟/模块** |

### 6. 怎么算做完？（验收标准）

- [ ] 代码能在微信开发者工具正常运行
- [ ] 核心功能流程跑通（能点、能播、能记录）
- [ ] UI不丑，至少"能用"
- [ ] 你亲自测试过，没有明显bug
- [ ] 代码有注释，你大概能看懂逻辑

**修正指令：**
- 功能不对 → "这个阶段应该只能前进不能后退，请修正"
- UI太丑 → "参考这个风格调整：[截图/描述]"
- 有bug → "点击XX后报错：[错误信息]，请排查"

### 7. 出错了怎么办？（失败排查）

| 问题 | 症状 | 解法 |
|------|------|------|
| AI理解错需求 | 做出来的和想的不一样 | 用"用户故事"格式重述：作为【用户】，我想要【功能】，以便【价值】 |
| 代码跑不通 | 报错一堆红字 | 把完整错误信息贴给AI，让它解释并修复 |
| 越改越乱 | 修A坏B，陷入死循环 | 回滚到上一个稳定版本，重新描述需求 |
| 想加功能 | MVP还没跑通就想做高级功能 | 写下来放进"后续版本"，先守住核心 |

### 8. 以后能自动吗？（升级路径）

| 阶段 | 触发方式 | 你的角色 |
|------|----------|----------|
| **手动** | 主动发起 | 描述需求 + 验收代码 + 决策方向 |
| **半自动** | 建立代码模板库 | 选择模板 + 调整参数 + 验收 |
| **全自动** | 接入自动化部署 | 定义目标 + 抽检结果 |

**止观开发飞轮：**
```
觉察（用户反馈/数据）→ 行动（快速迭代）→ 调整（优化方案）→ 反思（沉淀方法）→ 再次觉察
```

## 提示词对照：失败版 vs 成功版

**❌ 失败版**
> 帮我做个冥想小程序，要有引导语音和记录功能。

AI输出：
- 通用模板代码
- 功能大而全但都不深
- 没有阶段化设计
- 音频播放逻辑粗糙

**✅ 成功版**
> [使用上述完整模板，明确四阶段流程、Edge TTS音频、微信小程序平台]

AI输出：
- 针对四阶段流程的结构化代码
- 音频播放处理微信小程序限制
- 清晰的阶段跳转逻辑
- 可测试、可迭代的MVP

## 💡 高手心法

**你不是在"让AI写代码"，你是在"用AI做翻译"。**

把"我想做一个帮助人在碎片时间冥想的小程序"翻译成可执行的代码，中间隔着巨大的技术鸿沟。AI的价值是填平这个鸿沟，但前提是你能提供清晰的"源语言"——产品愿景、用户场景、功能边界。

宗国的「止止」能3天跑通MVP，不是因为他找到了一个"会写代码的AI"，而是因为他有八年的修行沉淀、清晰的场景定义、和敢于边做边学的行动力。

**记住一个公式：**
```
开发速度 = 愿景清晰度 × AI翻译能力 × 你的验收严格度
```

AI的翻译能力已经很强了。差距在于你的愿景够不够清晰、验收时敢不敢说不行。

**更深一层：借假修真**

开发「止止」的过程，本身就是一次止观练习。觉察需求（用户要什么）→ 行动实现（快速编码）→ 调整优化（修复bug）→ 反思沉淀（总结方法）。每一次循环，你的"开发操作系统"都在迭代。

这不仅是做一个小程序，而是在数字时代验证一条路径：技术可以是修行的助缘，而非干扰。
