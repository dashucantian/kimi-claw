# 卡 搭-X：用AI助手协作开发微信小程序

## 场景故事

2026年2月，产品经理小林想做一个正念冥想小程序。他懂产品逻辑，也写过一些代码，但完整的小程序开发对他来说是个黑箱——页面布局怎么写？音频怎么播放？数据怎么存储？

以前他可能会花两周啃文档、踩坑、调试。但这次他换了个思路：打开飞书，对着止观AI机器人说："我想做一个四阶段禅修的小程序，有音频引导、计时器和记录功能。"

3天后，止观小程序v1.0上线了。从需求描述到可运行的代码，大部分工作是在对话中完成的。小林负责描述需求和验证效果，AI负责生成代码和解释原理。

这不是"AI代替开发者"，而是"开发者有了个24小时在线的技术搭档"。

- 卡在具体技术点（如音频播放、本地存储）

### 2. 做出来是什么？给谁？（目标产出）

**产出物**：
- 可运行的微信小程序代码（HTML/CSS/JS）
- 项目文档和教程
- 可复用的开发方法论

**给谁用**：
- 产品经理/创业者：快速验证想法
- 初级开发者：边做边学
- 非技术背景创始人：降低开发门槛

### 3. 需要准备什么？（输入清单）

**必须**：
- 产品需求描述（功能列表、页面流程）
- 微信开发者工具（官方免费下载）
- 与AI助手的对话环境（如飞书机器人）

**建议**：
- 参考竞品截图或草图
- 音频素材（或让AI生成TTS）
- 设计稿或风格参考

**关键原则**：
需求描述越具体，AI输出越精准。不要只说"做个冥想小程序"，要说"四阶段流程：调身调息（2分钟）→止息（主要时长）→观心（1分钟）→收功（30秒）"。

### 5. 具体怎么操作？（步骤拆解）

| 步骤 | 动作 | 耗时 | 关键要点 |
|------|------|------|----------|
| 1 | 描述产品需求 | 10分钟 | 功能、流程、页面结构 |
| 2 | AI生成初版代码 | 5分钟 | 获取完整代码框架 |
| 3 | 导入开发者工具 | 3分钟 | 创建项目，粘贴代码 |
| 4 | 测试并记录问题 | 10分钟 | 功能、样式、交互 |
| 5 | 向AI反馈问题 | 5分钟 | 截图+描述 |
| 6 | AI修复并迭代 | 5分钟 | 获取修复代码 |
| 7 | 验证修复结果 | 5分钟 | 确认问题已解决 |
| 8 | 重复5-7直到满意 | - | 通常3-5轮 |

**总耗时**：首次搭建约2-3小时，后续迭代每轮10-15分钟

### 6. 怎么算做完？（验收标准）

- [ ] 首页可以正常选择时长和模式
- [ ] 练习页倒计时准确，阶段切换正常
- [ ] 音频可以正常播放和停止
- [ ] 完成页可以提交评分和备注
- [ ] 统计页显示历史记录
- [ ] 在微信开发者工具无报错
- [ ] 真机预览基本可用

**修正指令示例**：
- "倒计时在后台暂停了，需要保持运行" → 添加后台音频播放
- "底部导航被内容顶起来了" → 使用fixed定位+独立层级
- "音频无法循环播放" → 检查onEnded事件绑定

### 7. 出错了怎么办？（失败排查）

| 问题 | 症状 | 解法 |
|------|------|------|
| AI生成的代码跑不通 | 控制台报错 | 复制报错信息给AI；检查微信API版本兼容性 |
| 样式和预期差距大 | 布局错乱 | 提供参考截图；明确指定布局方式（flex/grid） |
| 音频无法播放 | 没声音或报错 | 检查音频文件路径；确认微信音频权限；尝试Web Audio API |
| 数据没有保存 | 刷新后记录丢失 | 检查storage调用；确认键名一致；添加存储成功回调 |
| 真机和模拟器表现不同 | 模拟器正常，真机异常 | 优先以真机为准；检查微信版本差异；使用真机调试 |

| 阶段 | 实现方式 | 你的角色 |
|------|----------|----------|
| 手动 | 每次对话描述需求 | 完整描述+验证 |

## 高手心法

**真正的效率提升，来自"对话式开发"的飞轮效应**

传统的开发流程是线性的：需求→设计→开发→测试→交付。每个环节之间是"扔过墙"的关系。

用AI助手协作开发，流程变成了循环的：

```
描述需求 → 生成代码 → 验证效果 → 反馈问题 → 修复迭代
     ↑                                      ↓
     └──────────── 循环优化 ────────────────┘
```

每一次循环，你都在做三件事：
1. **校准理解**：AI通过你的反馈，越来越懂你的需求
2. **沉淀知识**：遇到的问题和解决方案，成为项目文档
3. **提升表达**：你描述需求的能力越来越强

这就是「止观开发飞轮」的核心：
- **觉察**：快速看到当前版本的问题
- **行动**：立即反馈给AI生成修复
- **调整**：基于验证结果优化方向
- **反思**：记录经验，形成方法论

**记住一个公式**：

```
开发效率 = 需求清晰度 × AI理解准确度 × 验证反馈速度
```

AI的理解准确度已经很高了。差距在于你的需求描述是否足够具体，以及验证反馈是否足够及时。

当你觉得AI"不够聪明"时，先问自己三个问题：
1. 我描述的功能流程够清晰吗？
2. 我提供了足够的上下文吗？
3. 我反馈的问题够具体吗（有截图、有报错信息）？

这个方法不只适用于小程序开发。任何"你有想法但缺技术实现"的场景，都可以试试这个模式。

你不是在"让AI帮你写代码"，你是在训练一个懂你的技术搭档。

4. 文档和开发应该并行进行
